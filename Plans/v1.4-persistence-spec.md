## What we're building

A persistence layer behind the existing planner so Len can save, name, and reload customer records across sessions. Each customer record is the complete state of the planner — square footage, $/sqft, O&P, every line's budget, every paid amount. Len works on the planner exactly as today; the new behavior is **Save** and **Load**.

## The customer record model

Each record has:

| Field | Source | Notes |
|-------|--------|-------|
| **ID** | Auto-assigned, immutable | Zero-padded sequential — `0001`, `0002`, `0003`. Never reused, never edited. |
| **Description** | Len types it | Free text: "DONOVAN HOME", "Henderson Build", "Smith Renovation 2026". Optional but encouraged. Editable any time. |
| Created at | Auto, ISO timestamp | Set on first save. |
| Last updated at | Auto, ISO timestamp | Set on every save. |
| sqft | From planner | Customer's home square footage. |
| cost_per_sqft | From planner | Len's $/sqft for this build. |
| base_build_budget | From planner | Computed but stored for audit. |
| oop_pct | From planner | The O&P slider value. |
| total_project_budget | From planner | Audit field. |
| line_budgets[1..44] | From planner | One number per bank line. |
| line_paid[1..44] | From planner | One number per bank line. |

## Dropdown behavior (the part you specified)

The customer picker sits top-left in the planner, next to the LC logo. It always shows the auto-ID first, description second:

```
Select Customer ▾
─────────────────────────
+ New Customer
─────────────────────────
0001 — DONOVAN HOME
0002 — Henderson Build · Coweta County
0003 — Smith Renovation
0004 — (no description)
```

- "+ New Customer" at the top creates a fresh record, auto-assigns the next ID (e.g. `0005`), opens the description field with focus so Len types "JOHNSON FAMILY ROOM" before doing anything else.
- Existing records sort by **most recently updated**, descending — what Len touched yesterday floats to the top.
- The description field stays editable on every customer's record. Typo? Just click it and fix.

## Storage choice — Google Sheets

**Owner: Len's Google account.** His data, his ownership. You get a viewer invitation for support and demos.

Two sheets in one document called `LC Construction Budget — Customer Records`:

### Sheet 1: `customers` (index)

| col | name | type |
|-----|------|------|
| A | id | string `"0001"` |
| B | description | string |
| C | created_at | ISO timestamp |
| D | updated_at | ISO timestamp |
| E | sqft | number |
| F | cost_per_sqft | number |
| G | base_build_budget | number |
| H | oop_pct | number |
| I | total_project_budget | number |

One row per customer.

### Sheet 2: `line_items` (the detail)

| col | name | type |
|-----|------|------|
| A | customer_id | string `"0001"` |
| B | line_id | number 1..44 |
| C | budget_dollars | number |
| D | paid_dollars | number |

44 rows per customer. So Len's 50th customer pushes this sheet to ~2,200 rows — comfortably within Sheets limits (10M cells).

## Backend — Google Apps Script

Apps Script binds to the spreadsheet and exposes a single web app endpoint. Three operations, all JSON:

| Operation | Method | What it does |
|-----------|--------|--------------|
| `list` | GET | Returns the customer index (id, description, updated_at) for the dropdown |
| `load` | GET `?op=load&id=0042` | Returns the full record for one customer |
| `save` | POST | Upserts a customer record (creates if `id` is omitted, updates if present) |

Auto-ID generation is server-side: when `save` arrives with no `id`, Apps Script reads the highest existing ID, increments, zero-pads to 4 digits. Race-free because Apps Script serializes script execution per user.

## UI changes to the planner

Five additions, no removals:

1. **Customer picker** (top-left, next to the LC logo block) — dropdown with the format above.
2. **Description field** (right of the picker, appears when a customer is active) — single-line input, placeholder "Add a description for this record…", auto-saves on blur.
3. **Save button** (right of the description field) — only enabled when there are unsaved changes. Shows "Saving…" → "Saved" → fades to "Last saved 2:34 PM".
4. **Save indicator** (next to the Download PDF button) — passive "Last saved at 2:34 PM" or "Unsaved changes" with an amber dot when dirty.
5. **First-run banner** (only on first load when no sheet is connected yet) — small amber-tinted strip: "Connect your Customer Records spreadsheet to save jobs. [Connect]"

## Auto-save behavior

Editing a paid amount, budget, sqft, or any input marks the record dirty. Debounced 2 seconds — if Len's still typing, the save waits. On idle, the planner POSTs to Apps Script.

If the save fails (network down, Apps Script error), the indicator turns red "Save failed — retrying", and the data stays in localStorage as a backup. The next successful save flushes both.

## What happens if Apps Script is down

- Planner works fully in-memory exactly as today
- localStorage holds the current customer's state as a backup
- Save indicator shows "Offline — changes saved locally"
- When connection returns, planner auto-syncs

Len never loses work because the network blinked.

## Setup flow (one-time)

When Len first opens the planner:

1. Banner appears: "Connect your Customer Records spreadsheet to save jobs."
2. He clicks **Connect**.
3. Google OAuth consent screen pops up — "LC Planner wants to access your Google Sheets."
4. Len approves. (One time, ever.)
5. Apps Script auto-creates the spreadsheet `LC Construction Budget — Customer Records` in his Drive, sets up both sheets, returns the URL.
6. Planner stores the spreadsheet ID in localStorage. Future loads bypass step 4-5.

Alternative: Eugene creates the sheet manually first time, shares the URL with Len, planner remembers it. Skips OAuth. Tradeoff is Len doesn't own the file directly — you do.

**My recommendation:** Len owns the file. Slightly more setup friction, but the data sits in his Workspace where it belongs.

## Risks to surface before building

1. **Apps Script auth complexity.** First time Len authenticates, the consent screen says "This app isn't verified" because Google hasn't reviewed it. Workarounds: (a) publish under a Workspace-restricted scope so the warning is softer, (b) walk Len through the "Advanced → Go to LC Planner (unsafe)" click, (c) get the app verified by Google (free, takes ~2 weeks).

2. **Sheet ownership.** If Len ever loses access to his Google account, the data goes with him. Recommendation: have Eugene as an editor on the sheet so there's a backup path.

3. **Concurrent edits.** If Len is on his laptop and Eugene is in the sheet directly, last-write-wins. Not a real issue for solo-builder usage but worth noting.

4. **No multi-customer view yet.** The planner shows one customer at a time. If Len ever wants "show me all my jobs side by side," that's a Looker Studio dashboard, a separate v1.5.
